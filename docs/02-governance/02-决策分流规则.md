# 02 决策分流规则

## Type 0：运行中用户指令（Runtime Directive）

- 示例：用户临时纠偏、追加约束、要求暂停或取消某条工作线
- 处理：Orchestrator 创建 Directive，评估影响范围，并决定继续、暂停、取消、替换或重规划

## Type 1：执行失败（Execution Failure）

- 示例：命令报错、补丁冲突、测试失败
- 处理：Orchestrator 判断重试、重派、替换 Worker 或升级为 Issue

## Type 2：设计冲突（Design Conflict）

- 示例：模块边界冲突、架构约束冲突
- 处理：Orchestrator 评估影响，必要时形成候选方案并落出 Decision

## Type 3：需求冲突（Requirement Conflict）

- 示例：业务方向不清、范围扩张、取舍争议
- 处理：升级到 `Steering / Escalation Authority` 或用户确认

## 路由约束

- Worker 只能上报，不得自行做全局裁决。
- Orchestrator 负责分流、记录、升级。
- `Steering / Escalation Authority` 仅承担高阶裁决职责，不承担日常调度。

## 升级输出最小字段

- 冲突类型
- 触发任务
- 受影响 Directive / Task / Phase
- 影响范围
- 候选方案
- 推荐处理路径
